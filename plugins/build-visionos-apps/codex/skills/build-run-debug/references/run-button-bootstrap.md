# Run Button Bootstrap

This is the fallback bootstrap contract for the visionOS Build plugin's local
run loop.

Use this only when XcodeBuildMCP is not the chosen path for the task or when
the user explicitly wants a persistent Codex `Run` button.

Before bootstrapping, confirm you already attempted the MCP-first sequence:

1. `session-show-defaults`
2. `list_sims`
3. `session-set-defaults`
4. `build_run_sim` or `build_sim` for the current goal

Only generate the shell path when that sequence is unavailable or when the user
explicitly requests a persistent Run action.

When a project does not already have an established visionOS run entrypoint:

1. Create one project-local `script/build_and_run.sh`.
2. Make it executable.
3. Use it as the single build + install + launch entrypoint.
4. Support `--debug`, `--logs`, `--telemetry`, and `--verify`.
5. Write `.codex/environments/environment.toml` so the Codex app exposes a
   `Run` action wired to that script.

## Plugin Bootstrap Helper

Use the packaged helper script from the plugin root to generate the local run
loop:

```bash
plugins/build-visionos-apps/scripts/bootstrap_build_and_run.sh \
  --project MyApp.xcodeproj \
  --scheme MyApp
```

The helper supports these flags:

| Flag | Purpose |
|------|---------|
| `--project <path>` | Path to an `.xcodeproj`. Mutually exclusive with `--workspace`. |
| `--workspace <path>` | Path to an `.xcworkspace`. Mutually exclusive with `--project`. |
| `--scheme <name>` | Scheme to build. Required. |
| `--simulator-name <name>` | Pin the generated script to a specific simulator name (for example `"Apple Vision Pro"`). Optional. |
| `--bundle-id <id>` | Pin the app bundle identifier. Optional — derived from the built `.app` when not set. |
| `--derived-data <path>` | Override DerivedData location. Optional. |
| `--overwrite` | Replace an existing project-local `script/build_and_run.sh` and/or `.codex/environments/environment.toml` `Run` action. Without this flag the helper refuses to clobber existing files. |

The helper generates:

- `script/build_and_run.sh`
- `.codex/environments/environment.toml`

## `script/build_and_run.sh`

Use one project-specific script with a default no-flag path that:

1. resolves or boots an available Apple Vision Pro simulator,
2. builds the chosen scheme with `xcodebuild`,
3. installs the built `.app` into the simulator,
4. terminates any existing instance,
5. launches the new build.

The script should use `xcodebuild`, `xcrun simctl`, and a deterministic
DerivedData path inside the project, not one-off manual terminal commands. It
should mirror the scheme and simulator choice already resolved through the MCP
or repo inspection workflow.

The script is a fallback runtime wrapper, not the source of truth for scheme or
destination selection when MCP is available.

### Expected flags

- `run`: build, install, and launch
- `--debug`: build, install, and launch while waiting for debugger attach
- `--logs`: build, install, and launch with console logging
- `--telemetry`: build, install, and launch, then stream unified logs for the app subsystem
- `--verify`: build, install, and launch, then confirm a launch pid was returned

## `.codex/environments/environment.toml`

Write the environment file at this exact path:

`.codex/environments/environment.toml`

with this action shape:

```toml
# THIS IS AUTOGENERATED. DO NOT EDIT MANUALLY
version = 1
name = "<project-name>"

[setup]
script = ""

[[actions]]
name = "Run"
icon = "run"
command = "./script/build_and_run.sh"
```

If the project already has an environment file, update the existing `Run`
action to point at `./script/build_and_run.sh` instead of adding a duplicate.

If the environment already points to this script path, do not rewrite it unless
the user asks for a behavior change.
